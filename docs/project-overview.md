# SmartMate 项目功能说明

> **迁移状态**：纯 `.h/.cpp` Qt Widgets 已完成迁移并成为唯一正式前端；迁移未改变本文的业务规则、数据格式、原子事务或功能语义。

## 1. 项目定位

SmartMate 是一款本地、单用户、仅面向 Windows 的桌面任务规划应用。它不仅记录“要做什么”，还根据任务依赖、优先级和截止时间回答“现在应该先做什么”。桌面宠物作为主窗口之外的第二个 View，以更轻量、更有陪伴感的方式展示和操作当前任务。

本项目同时是一项 MVVM 架构实践。功能设计优先服务于一条完整、可解释、可测试的数据链路，而不是追求大量彼此孤立的功能。

当前版本已完成任务管理、用户自定义类别与 Finish-to-Start 依赖的纵向链路：Widget 只通过抽象 Contract 向 Task ViewModel 转发操作，Model Service 统一执行类别校验、显式状态转换、编辑资格、批量归档恢复、永久删除、循环检测、阻塞解锁和自动排序，SQLite Repository 负责持久化。活动任务采用软归档；只有归档任务可以确认后永久删除，并同时清理关联依赖。界面支持实时搜索、优先级与类别筛选、多选批量管理、逾期提醒、创建或编辑前置关系、阻塞原因、推荐理由及分类有向依赖图展示。

当前主窗口使用左侧导航依次组织任务、依赖图、专注、统计和设置。任务页顶部的“现在做”槽独立于搜索和筛选：优先展示当前进行中任务，否则淡化展示 Model 推荐的第一项可执行任务；用户也可通过专用拖动手柄把任意可执行卡片拖入槽中开始。卡片点击打开只读详情，状态变化仍只能通过显式命令完成。

外观设置提供青绿与清蓝强调色、有限字体和三档字号。选择经 `AppearanceSettingsViewModel`、Model Service 与外观设置 Repository 保存，具体 `QSettings` 调用只位于 persistence；主题颜色和像素字号由 View 层令牌解释。

任务编辑器通过日历、时钟以及天/小时/分钟选择器收集截止时间和预计用时，不要求用户输入固定格式文本。Widget 只转发结构化选择，ViewModel 负责本地时间与展示草稿，Model 仍执行最终业务校验。

这条链路刻意把职责拆开：普通 C++ `Task` 不知道界面如何显示；`TaskService` 不知道数据来自 SQLite 还是测试 Fake；ViewModel 不知道 SQL 表结构；Widget 也不重新实现标题校验和状态规则。因此每一层都能独立测试，桌宠也只增加新的展示入口，而没有复制任务业务逻辑。

界面主题不是“灰白界面加一个强调色”，而是由 View 层 `WidgetTheme` 统一解释完整的背景、导航、表面、增强表面、输入背景、边框及交互状态令牌。青绿与清蓝切换时所有主要表面一起变化，任务状态、警告和危险色则保持独立语义。正文使用与主题协调的深青灰或深蓝灰，避免浅色文字造成阅读负担。

任务卡片拖放只通过内部 MIME 暴露稳定 TaskId。释放到“现在做”槽后，Widget 只转发一次开始命令，最终是否允许开始仍由 `TaskService` 检查前置依赖和单进行中约束。拖放不表示列表排序，阻塞任务不提供拖动入口。

依赖图使用自上而下的分层任务地图：Model 提供最长前置链层级和上下游闭包，ViewModel 通过稳定同层排序、虚拟路由点、分散端口和层间通道生成正交路径，`QGraphicsView/QGraphicsScene` 只负责绘制。选择节点后完整前置链与后继链被突出，无关内容淡化；右侧详情采用响应式左右并排布局，在最小窗口下也不会覆盖依赖图画布。

任务编辑器按基本信息、任务规划、时间与依赖分区，宽度随窗口收缩；内容区固定为可用宽度且只允许纵向滚动。日期、时长、前置任务和依赖编辑弹窗沿用相同主题与响应式边界，确保最小窗口和较大字号下不会产生水平溢出。

## 2. 核心使用流程

1. 用户创建任务，填写标题、描述、优先级、截止时间和预计用时，并可在保存前选择前置任务；任务状态固定为待办。
2. 系统在同一事务中保存任务与全部前置关系，或在任意失败时完整回滚。
3. 系统区分可执行与被阻塞任务，并生成稳定、可解释的推荐顺序。
4. 用户选择一个任务开始执行；任意时刻最多只有一个进行中的任务。
5. 主窗口、桌宠和专注计时同步显示当前任务。
6. 用户完成或取消前置任务后，系统立即解除符合条件的后继任务并重新规划；取消任务被重做后，原依赖自动恢复。
7. 用户可在有向依赖图中查看执行链、阻塞箭头和节点详情。
8. 统计页面先从任务事件汇总完成成果和当前任务健康；专注模块完成后再接入专注会话数据。

## 3. 必须完成的功能

### 3.1 任务管理

任务包含以下核心信息：

- 稳定且唯一的任务 ID；
- 标题与描述；
- 状态：待办、进行中、已完成、已取消、已归档；状态不属于可编辑草稿；
- 优先级；
- 截止时间与预计用时；
- 创建时间与最后修改时间。

支持的操作包括：

- 新建和查看任务，新建时可一并选择现有活动任务作为前置；
- 只有待办任务可以使用编辑草稿修改普通字段，取消编辑时不污染原数据；其他状态必须先通过合法命令回到待办；
- 使用日历与固定时间选择器设置可选截止时间和预计用时；
- 按显式状态机开始、取消、完成和重做任务：待办只能开始或取消，进行中只能完成或取消，完成/取消只能重做为待办；
- 只有完成或取消任务可归档；归档任务恢复到原完成/取消状态，旧数据库中的异常恢复目标统一降级为待办；
- 可按当前搜索与优先级筛选结果多选任务，整批原子归档或恢复；活动视图只允许勾选具备归档资格的任务；
- 归档任务可逐项或批量永久删除；删除会在同一事务中清理全部前后依赖且不可撤销；
- 对空标题、非法时间等输入给出明确错误。

“阻塞”“可执行”“逾期”是根据当前数据动态计算的派生状态，不作为容易失真的字段写入数据库。只有待办和进行中任务在截止时间早于当前UTC时间时标记逾期；完成、取消与归档任务不再提示。

状态转换表固定如下，界面按钮只展示ViewModel根据同一领域规则投影出的可用命令：

| 命令 | 允许来源 | 目标 | Service附加校验 |
| --- | --- | --- | --- |
| 开始 | 待办 | 进行中 | 前置关系均已完成或取消，且没有其他进行中任务 |
| 取消 | 待办、进行中 | 已取消 | 允许被阻塞任务取消，后继关系立即暂时失效 |
| 完成 | 进行中 | 已完成 | 再次验证前置关系，禁止待办直接完成 |
| 重做 | 已完成、已取消 | 待办 | 重新激活原依赖，不得破坏已开始或已完成后继 |
| 归档 | 已完成、已取消 | 已归档 | 保存归档前终态，不删除任务或依赖边 |
| 恢复 | 已归档 | 原完成/取消状态 | 旧版异常恢复点统一归一为待办 |

把状态从编辑字段改为显式命令，能够让Model完整表达工作流，并避免不同View通过下拉框制造“待办直接完成”等非法快照。ViewModel无需复制转换表，只负责把Model给出的资格映射成按钮可见性；未来桌宠使用同一命令时也会获得完全一致的规则。

### 3.2 查询、筛选与排序

- 实时按标题或描述关键字搜索；
- 在当前活动或归档视图中筛选全部、低、普通、高或紧急优先级；
- 由 Model 根据任务状态、逾期情况、优先级和截止时间生成稳定顺序与推荐理由；
- 搜索和优先级条件相互叠加，清除条件后恢复当前活动或归档视图的完整结果；
- 所有操作都使用任务 ID，而不是界面列表行号识别任务。

自动排序规则属于 Model；搜索和优先级筛选只改变 ViewModel 提供的展示投影。搜索词、筛选条件和排序结果都不修改任务，也不写入 SQLite 或 `QSettings`，重新启动应用后恢复默认条件。若后续增加“可执行/被阻塞”筛选，也只能过滤 Model 已计算的依赖状态。

### 3.3 任务类别

- 每个任务最多归入一个用户自定义类别，也可以保持未分类；类别使用稳定ID，名称和颜色不作为身份。
- 类别支持创建、重命名、固定调色板换色和删除，首次启动不自动创建示例数据。
- 只有待办任务可以主动调整类别；删除类别时所有关联任务在同一事务中转为未分类，状态和依赖保持不变。
- 列表中的类别条件与关键字、优先级采用AND组合；列表页和依赖图页分别保留自己的会话筛选条件。
- 跨类别依赖合法。分类依赖图显示类别核心节点及其直接跨类别前置、后继上下文，不递归展开整个外部链路。

类别规则和图节点裁剪属于 Model，ViewModel 只维护类别草稿、筛选条件和展示投影；Widget 必须通过稳定 `TaskCategoryId` 转发操作，不得使用名称或行号识别类别。

### 3.4 前后依赖关系

第一版实现清晰、实用的Finish-to-Start关系。例如 A 是 B 的前置任务，则关系处于待满足时 B 不能开始或完成。多个前置关系采用AND语义，每条关系必须已完成或已取消：完成及归档前完成表示永久满足；取消及归档前取消表示关系暂时失效且不阻塞。取消不会删除SQLite中的边，A被重做为待办后原关系自动重新生效。

系统必须拒绝：

- 任务依赖自身；
- 完全重复的依赖；
- 会形成环的依赖，例如已有 `A → B → C` 时再添加 `C → A`。

只有活动待办任务可以修改自己的前置集合。已取消任务不能新增为前置候选；已经存在的取消关系仍须显示并允许移除。已有关系在任务归档后继续保留；添加、移除、完成、取消、恢复或重做任务后，系统都会重新计算直接阻塞原因、可解锁数量和推荐顺序。阻塞任务不能开始；任务不得从待办直接完成。系统也会拒绝重做某个前置任务而使已进行中、已完成或归档前属于这些状态的后继重新失效。

携带前置关系的新任务只能以待办状态创建。创建表单中的候选多选与其他字段属于同一个草稿；点击保存后，SQLite在一个事务内插入任务和所有依赖边，任一约束或写入失败都不会留下半成品任务。

### 3.5 有向依赖图

- 从左向右按最长前置链层级排列节点，用箭头表示“前置 → 后继”；
- 显示全部活动任务、活动任务所在链中的归档节点和无依赖的活动任务，隐藏纯归档组件；
- 已完成关系显示绿色箭头，未满足关系显示橙色箭头，已取消关系显示灰色箭头；阻塞、归档和选中节点使用不同边框；
- 支持画布平移、0.5～2.0倍缩放、适应窗口、恢复100%和节点详情；
- 可从活动待办节点进入现有依赖编辑器，但不允许直接拖线或删除边。

这里的职责拆分能够直接演示 MVVM 优势：Model 只计算拓扑层级、连接闭包和满足状态；`TaskGraphViewModel` 只计算节点坐标、正交路径和箭头三角形；Widget 只按 Role 绘制并转发稳定任务 ID。任何图布局和选中状态都不写入数据库。

### 3.6 自动规划

当前版本的Model为每个任务生成依赖感知的稳定位置和简短理由，例如“正在进行”“已逾期”“紧急优先级”或“即将到期”；ViewModel只将结构化结果映射为界面文本，并在排序结果上应用会话级搜索与筛选。

规划器先排列进行中与当前可执行的待办任务，再使用拓扑顺序整理被阻塞任务；同一可执行集合继续比较：

1. 是否已经逾期；
2. 优先级；
3. 截止时间；
4. 稳定的任务 ID，用于确保相同输入始终得到相同结果。

规划结果同时提供简短理由，例如“已逾期”“高优先级”“即将到期”或“完成或取消后可解锁 2 个任务”。这样推荐是可解释的，而不是不可验证的黑盒评分。

## 4. 桌面宠物

桌宠是 SmartMate 的创新展示形式，也是验证 MVVM 多 View 同步的重要功能。首版现已实现，完整窗口行为、设置持久化、动画资产和验收口径见[三花猫桌宠设计](desktop-pet-design.md)。当前版本包括：

- 设置页提供默认关闭的桌宠开关；
- 普通非最大化窗口显示靠右、鼠标穿透的趴卧桌宠，最大化与 Qt 全屏时隐藏；
- 主窗口最小化后切换为无边框、透明、置顶且可拖动的悬浮坐姿桌宠；
- 以屏幕名称和归一化坐标记住悬浮位置，并在显示器变化后修正到可见区域；
- 显示当前进行中的任务，或推荐的下一个可执行任务；
- 快捷开始、完成任务和打开主窗口；
- 没有可执行任务时区分“全部完成”和“任务均被阻塞”。

桌宠不直接调用任何具体 ViewModel。任务展示复用 `TaskFocusContract`，开始与完成命令复用 `TaskListContract`，设置开关与悬浮位置使用独立 `DesktopPetSettingsContract`；主窗口修改任务后桌宠自动更新，桌宠操作后主窗口也通过现有 Service 失效链路自动刷新。推荐、阻塞、资格和状态转换继续由 Model 计算，Widget 不得复制规则。

首版角色固定为像素风“三花猫”：悬浮状态使用坐姿 `idle` 动画，窗口附着状态使用趴卧 `waiting` 动画。后续可增加完成庆祝、命令失败和专注查看等动作；帧播放、锚点和透明绘制只属于 View，任何任务语义仍由 Model 与 ViewModel Contract 提供。

## 5. 专注计时与统计

任务统计的事件、Schema v4、Model 聚合服务、Statistics Contracts、具体 ViewModel、纯 Widgets 仪表盘、主窗口导航、组合根注入与纵向自动刷新已经完整实现。专注功能的 Schema v5、领域事实、Repository、`FocusService`、检查点、启动/退出恢复、任务状态保护、Contracts、`FocusViewModel`、纯 Widgets 页面、导航和纵向自动刷新也已完成；专注统计仍待后续阶段。统计口径见[数据统计设计](statistics-design.md)，专注阶段与边界见[专注功能设计](focus-design.md)。

### 5.1 专注计时

- 只有进行中的任务才能开始自由正计时专注，一个任务可以完成多次会话；
- 任意时刻最多一个活动会话；
- 暂停期间不累计时间；
- 可以继续、完成或放弃会话；放弃会话物理删除且不进入历史；
- 通过多个原始时间段汇总实际专注时长，不在数据库保存重复总时长；
- 专注页显示当前会话与秒级累计时间；任务页和桌宠继续共享同一当前任务投影，不复制专注计时器。

当前 `FocusService` 以 UTC 汇总原始时间段，保存检查点、任务标题、预计用时、类别和可空任务 Start 事件快照。累计不足一秒不能完成；应用退出或异常恢复后不足一秒的活动会话自动放弃。Running 与 Paused 都会阻止关联任务完成或取消，最终保护位于同一 SQLite 状态事务。`FocusViewModel` 只把 Service 给出的候选、资格和累计值格式化为秒级计时与历史投影，不复制状态机。正式专注页位于主窗口索引 2，无进行中任务时可返回任务页，放弃前必须确认且不会保留记录。

### 5.2 数据统计

第一版任务统计采用“一眼看结果、继续看趋势、最后找原因”的仪表盘：

- 顶部显示今日完成、本周完成、同期变化、本周按时完成率和当前逾期；
- 完成趋势支持近 7 天、近 30 天和近 12 周，前两项按本地自然日、后一项按周一开始的本地自然周分桶；
- 当前任务健康展示可执行、被阻塞、即将到期和已经逾期；
- 所选周期按完成时类别展示成果构成，保留前五个类别并把其余合并为“其他”。

每次合法 `Complete` 转换记为一次完成；任务重做后再次完成应再次计数，归档和恢复不计数。按时率只统计具有截止时间快照的完成事件，没有截止时间的事件不进入分母。今日和本周分别与昨日、上周相同已过去时长比较，避免用未结束周期对比完整周期。

完成趋势以 Schema v4 任务事件为事实来源。任务状态更新和事件插入在同一 SQLite 事务中提交；Repository 只返回 UTC 原始事实，Model 根据显式时区计算本地自然日、趋势、同期变化、按时率和任务健康。数据库不保存每日、每周等重复汇总值。

Schema v3 的旧任务没有可靠完成时间，迁移时不使用 `updatedAtUtc` 推测或回填历史；当前任务健康仍包含旧任务，完成趋势从任务事件功能启用后开始。永久删除归档任务时同步级联删除其任务事件，统计结果随之变化。

统计页面保持纯 Qt Widgets：数字卡和任务健康使用普通控件，完成趋势使用 Qt Charts `QBarSeries`，类别构成使用 `QHorizontalBarSeries`，按时率使用 `QPieSeries` 环形图。Qt Charts 只属于 View，禁止引入 Qt Graphs、`QQuickWidget`、QML 或 Qt Quick，也禁止让 Charts 类型进入 ViewModel Contracts、具体 ViewModel、Model 或 Persistence。

后续增加今日/本周专注时长、独立专注趋势和预计用时与实际专注用时对比；不同单位不得混入同一双 Y 轴图表。

## 6. 数据持久化

正式程序当前使用 SQLite Schema v5 保存任务、类别、依赖关系、不可变任务事件以及专注会话/时间段，使用 `QSettings` 保存窗口位置等轻量偏好。迁移不为旧任务回填专注记录，也不修改已有任务或任务事件。Model 只依赖 Repository 接口，SQLite 具体实现仅位于 `src/model/persistence`。

这一设计允许单元测试使用内存 Fake Repository，也允许未来替换数据库而不修改 ViewModel 和 Widget。

## 7. 后续扩展

以下功能本轮不承诺实现，但架构为其保留扩展点：

- 可拖拽连线、切换依赖类型的高级图编辑；
- Windows 系统通知与托盘常驻；
- 更丰富的桌宠动画、换装和反馈；
- 日历、重复任务、标签、子任务与模板；
- 数据备份与导入导出；
- 更丰富的统计图表。

这些功能必须在核心闭环、测试、文档和发布打包都稳定后再加入，不能以破坏 MVVM 边界为代价。

## 9. 最能体现 MVVM 的演示场景

1. 添加会形成环的依赖，Model 拒绝操作并由 ViewModel 映射为界面错误。
2. 新建任务时选择前置任务，任务与关系要么同时出现，要么失败后都不出现。
3. 完成或取消一个前置任务，后继立即解锁且箭头同步变为绿色或灰色；重做取消任务后原边重新生效。
4. 同一份任务状态与事件事实同时被任务列表、依赖图、桌宠和统计页以不同形式展示。
5. 使用 Fake Repository 在不启动 GUI 和 SQLite 的情况下测试任务服务与规划算法。

这些场景能清楚说明 MVVM 的核心优势：业务规则独立、依赖可替换、界面可测试、多个 View 可共享状态且彼此解耦。
