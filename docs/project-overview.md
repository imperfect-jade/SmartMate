# SmartMate 项目功能说明

## 1. 项目定位

SmartMate 是一款本地、单用户、仅面向 Windows 的桌面任务规划应用。它不仅记录“要做什么”，还根据任务依赖、优先级和截止时间回答“现在应该先做什么”。桌面宠物作为主窗口之外的第二个 View，以更轻量、更有陪伴感的方式展示和操作当前任务。

本项目同时是一项 MVVM 架构实践。功能设计优先服务于一条完整、可解释、可测试的数据链路，而不是追求大量彼此孤立的功能。

当前版本已完成任务管理的第一条纵向链路：简单 QML 界面只向 Task ViewModel 转发操作，`TaskService` 统一执行校验、单进行中约束、归档恢复和基础自动排序，SQLite Repository 负责持久化。任务采用软归档，不提供永久删除；恢复时回到归档前状态。界面已支持实时搜索、优先级筛选和推荐理由展示。

任务编辑器通过日历、时钟以及天/小时/分钟选择器收集截止时间和预计用时，不要求用户输入固定格式文本。QML只转发结构化选择，ViewModel负责本地时间与展示草稿，Model仍执行最终业务校验。

这条链路刻意把职责拆开：普通 C++ `Task` 不知道界面如何显示；`TaskService` 不知道数据来自 SQLite 还是测试 Fake；ViewModel 不知道 SQL 表结构；QML 也不重新实现标题校验和状态规则。因此每一层都能独立测试，未来增加桌宠时也只需增加新的展示投影，而不用复制任务业务逻辑。

## 2. 核心使用流程

1. 用户创建任务，填写标题、描述、优先级、截止时间和预计用时。
2. 用户为任务添加前置关系，例如“完成需求分析后才能开始编码”。
3. 系统排除仍被阻塞的任务，并生成稳定、可解释的推荐顺序。
4. 用户选择一个任务开始执行；任意时刻最多只有一个进行中的任务。
5. 主窗口、桌宠和专注计时同步显示当前任务。
6. 用户完成任务后，系统立即解除符合条件的后继任务并重新规划。
7. 统计页面从任务和专注记录中汇总进度，帮助用户复盘。

## 3. 必须完成的功能

### 3.1 任务管理

任务包含以下核心信息：

- 稳定且唯一的任务 ID；
- 标题与描述；
- 状态：待办、进行中、已完成、已取消、已归档；
- 优先级；
- 截止时间与预计用时；
- 创建时间与最后修改时间。

支持的操作包括：

- 新建和查看任务；
- 使用编辑草稿修改任务，取消编辑时不污染原数据；
- 使用日历与固定时间选择器设置可选截止时间和预计用时；
- 开始、完成、取消和重新打开任务；
- 归档与恢复任务，避免直接物理删除造成依赖关系丢失；
- 对空标题、非法时间等输入给出明确错误。

“阻塞”“可执行”“逾期”是根据当前数据动态计算的派生状态，不作为容易失真的字段写入数据库。

### 3.2 查询、筛选与排序

- 实时按标题或描述关键字搜索；
- 在当前活动或归档视图中筛选全部、低、普通、高或紧急优先级；
- 由 Model 根据任务状态、逾期情况、优先级和截止时间生成稳定顺序与推荐理由；
- 搜索和优先级条件相互叠加，清除条件后恢复当前活动或归档视图的完整结果；
- 所有操作都使用任务 ID，而不是界面列表行号识别任务。

自动排序规则属于 Model；搜索和优先级筛选只改变 ViewModel 提供的展示投影。搜索词、筛选条件和排序结果都不修改任务，也不写入 SQLite 或 `QSettings`，重新启动应用后恢复默认条件。依赖关系加入后，再扩展可执行、被阻塞等领域筛选。

### 3.3 前后依赖关系

第一版只实现清晰、实用的“完成后才能开始”（Finish-to-Start）关系。例如 A 是 B 的前置任务，则 A 完成前 B 不能开始。

系统必须拒绝：

- 任务依赖自身；
- 完全重复的依赖；
- 会形成环的依赖，例如已有 `A → B → C` 时再添加 `C → A`。

被取消的前置任务不视为已完成，因此后继任务仍保持阻塞。添加或删除关系、完成或重新打开任务后，都必须重新计算阻塞状态和推荐顺序。

### 3.4 自动规划

当前版本已经实现不含依赖关系的基础自动排序。Model 为每个任务生成稳定位置和简短理由，例如“正在进行”“已逾期”“紧急优先级”或“即将到期”；ViewModel 只将理由映射为界面文本，并在排序结果上应用会话级搜索与筛选。

加入依赖关系后的增强规划器将首先使用拓扑排序保证所有前置任务都排在后继任务之前，再只在当前可执行任务之间比较：

1. 是否已经逾期；
2. 优先级；
3. 截止时间；
4. 稳定的任务 ID，用于确保相同输入始终得到相同结果。

规划结果同时提供简短理由，例如“已逾期”“高优先级”“即将到期”或“完成后可解锁 2 个任务”。这样推荐是可解释的，而不是不可验证的黑盒评分。

## 4. 桌面宠物

桌宠是 SmartMate 的创新展示形式，也是验证 MVVM 多 View 同步的重要功能。最小可交付版本包括：

- 无边框、透明、置顶窗口；
- 拖动并记住桌面位置；
- 显示当前进行中的任务，或推荐的下一个可执行任务；
- 快捷开始、完成任务和打开主窗口；
- 没有可执行任务时区分“全部完成”和“任务均被阻塞”。

桌宠不直接调用任务列表 ViewModel。`PetViewModel` 从同一 Model 服务生成适合桌宠的投影，因此主窗口修改任务后桌宠自动更新，桌宠操作后主窗口也自动更新。

时间允许时，可增加 Idle、Working、Urgent、Celebrate 四种视觉状态及简单帧动画。动画只属于 View；“是否紧急”“是否庆祝”等语义由 ViewModel 提供。

## 5. 专注计时与统计

这两个模块在基础任务和桌宠稳定后实现。

### 5.1 专注计时

- 为当前任务开始、暂停、继续和结束专注会话；
- 任意时刻最多一个活动会话；
- 暂停期间不累计时间；
- 结束后保存实际专注时长；
- 主窗口与桌宠显示一致的计时状态。

计时规则和时间计算属于 Model，ViewModel 只将剩余时间、按钮可用性和状态文本提供给界面。

### 5.2 数据统计

- 今日与本周完成任务数；
- 今日与本周累计专注时长；
- 任务预计用时与实际专注用时的对比；
- 可选的连续完成天数或任务完成率。

统计数据从任务和专注记录聚合得到，不保存容易与原始记录不一致的重复汇总值。第一版优先使用数字卡片和简单趋势，不投入时间开发复杂图表。

## 6. 数据持久化

正式程序使用 SQLite 保存任务、依赖关系和专注记录，使用 `QSettings` 保存窗口位置等轻量偏好。Model 只依赖 Repository 接口，SQLite 具体实现仅位于 `src/model/persistence`。

这一设计允许单元测试使用内存 Fake Repository，也允许未来替换数据库而不修改 ViewModel 和 QML。

## 7. 后续扩展

以下功能本轮不承诺实现，但架构为其保留扩展点：

- 可交互的任务依赖图；
- Windows 系统通知与托盘常驻；
- 更丰富的桌宠动画、换装和反馈；
- 日历、重复任务、标签、子任务与模板；
- 数据备份与导入导出；
- 更丰富的统计图表。

这些功能必须在核心闭环、测试、文档和发布打包都稳定后再加入，不能以破坏 MVVM 边界为代价。

## 8. 七天开发阶段

| 阶段 | 目标 | 完成标准 |
| --- | --- | --- |
| 第 1～2 天 | Task 领域模型、Repository、SQLite、CRUD、搜索筛选 | 能新增、编辑、归档、恢复，重启后数据一致 |
| 第 3 天 | 依赖规则、环检测、阻塞状态、规划算法 | 领域测试通过，依赖顺序不会被优先级破坏 |
| 第 4 天 | 主窗口集成、推荐理由、错误反馈 | 基础任务闭环稳定并冻结功能 |
| 第 5 天 | 桌宠与 `PetViewModel` | 主窗口和桌宠双向操作后自动同步 |
| 第 6 天 | 专注计时；有余力再做统计 | 中午冻结新功能，开始完整回归 |
| 第 7 天 | 修复、Release 部署、文档和答辩准备 | 在无 Qt 开发环境依赖下运行发布目录并完整演示 |

## 9. 最能体现 MVVM 的演示场景

1. 添加会形成环的依赖，Model 拒绝操作并由 ViewModel 映射为界面错误。
2. 完成一个前置任务，后继任务立即解锁，推荐顺序、主窗口和桌宠同步更新。
3. 同一份 Model 数据同时被任务列表、桌宠和统计页以不同形式展示。
4. 使用 Fake Repository 在不启动 GUI 和 SQLite 的情况下测试任务服务与规划算法。

这四个场景能清楚说明 MVVM 的核心优势：业务规则独立、依赖可替换、界面可测试、多个 View 可共享状态且彼此解耦。
